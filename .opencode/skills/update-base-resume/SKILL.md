# Update Base Resume with New Experiences

将用户提供的新经历、项目、技能或证书合并到 `content/page/about/index-base.md` 中，生成更新后的基础简历。

## 前置文件

| 文件                               | 用途                                                                         |
| ---------------------------------- | ---------------------------------------------------------------------------- |
| `content/page/about/index-base.md` | **基础简历（唯一事实来源）**，包含所有工作经历、项目、技术栈、证书等完整信息 |

## 适用边界

- 只在用户明确提供新的真实经历、项目、技能、证书或个人信息，并要求更新基础简历时使用。
- 如果用户只是提供 JD、岗位要求或要求生成针对岗位的简历，使用 `tailor-resume`，不要修改 `index-base.md`。
- `index-base.md` 是后续所有定制简历的事实源；更新时必须优先保证事实准确、可追溯、不过度包装。
- 用户没有明确提供的数据、年限、成果指标、证书编号、项目职责，不要推断或编造。

## 工作流程

### 第一步：读取现有基础简历

读取 `content/page/about/index-base.md`，完整理解现有结构：

- 个人基础信息
- 每段工作经历的公司、时间、角色、项目详情
- 技术方向与技能等级
- 证书与荣誉
- 博客与开源
- 其他特长

### 第二步：理解新经历

分析用户提供的新经历内容，识别需要更新的部分：

- **新增工作经历**：新的公司、岗位、时间、项目
- **现有经历的补充**：对已有项目的补充细节、新数据
- **新增技能/技术栈**：新掌握的语言、框架、工具
- **新增证书/认证**：新获得的证书
- **更新的个人信息**：时间范围更新（如"至今"的截止时间）、联系方式变更等

同时将信息分为三类：

- **可直接写入**：用户明确给出且与现有内容不冲突的信息
- **需要确认**：时间范围、公司/项目归属、量化指标、证书编号、职责边界不完整或含糊的信息
- **暂不写入**：推测性表达、与现有内容冲突且用户未确认的信息、明显属于岗位定制包装而非基础事实的信息

### 第三步：智能合并

根据新经历的类型执行对应的合并策略：

#### A. 新增整段工作经历

- 按时间倒序插入到 `## 工作经历` 部分（最新经历放在最前面）
- 格式与现有条目一致：

  ```markdown
  ### [公司名]｜[岗位]｜[方向/部门]

  - 时间：[起始年月] 至 [结束年月]
  - 部门 / 方向：[部门]、[业务方向]
  - 主要职责：[概括职责]

  #### [项目名]

  - [具体工作内容，每行一条，用 - 开头]
  - ...
  - 技术栈：[语言]、[框架]、[工具]...
  ```

#### B. 补充现有经历中的项目

- 定位到对应的公司和项目
- 将新内容以新要点的形式追加到现有列表中
- 更新技术栈（如果引入了新的技术关键词）
- 更新量化数据（仅限用户明确提供或现有基础简历已记录的数据指标）
- 如果新内容改变了原有职责、成果归属或时间范围，先询问用户确认，再修改

#### C. 更新技术方向

- 检查 `## 技术方向` 中是否有新方向需要添加
- 每个方向一行，格式：`- 方向名：具体技术点，用顿号或逗号分隔`
- 如有需要，扩展现有方向的说明

#### D. 更新技术栈

- 检查 `## 技术栈` 列表中是否需要新增条目
- 按分类添加：编程语言、后端框架、数据库、中间件、大数据、云原生/DevOps、AI/LLM/Infra
- 保持字母顺序或重要性排序的一致性

#### E. 更新证书与荣誉

- 新增证书追加到 `## 证书与荣誉` 列表末尾
- 格式：`- [证书名称](链接)（编号等附加信息）` 或 `- 证书名称：说明`

#### F. 更新个人简介（## 关于我）

- 如果新增经历代表了新的职业方向，更新第一段概述，但保持基础事实描述，不写成某个 JD 的定制卖点
- 调整年限描述（如"X年经验"），必须基于工作经历时间线计算或用户明确说明
- 如新增了重要的方向关键词，补充到概述中

## 合并原则

1. **不删除现有内容**：除非用户明确要求，否则保留所有现有经历。如有过期或冲突信息，询问用户确认后再修改
2. **保持格式一致**：严格遵循 `index-base.md` 的 Markdown 层级和样式（H2 用 `##`、H3 用 `###`、H4 用 `####`，列表用 `-`）
3. **时间线准确**：确保所有时间范围一致，无逻辑矛盾
4. **技术术语准确**：保持技术名词的大小写和拼写与现有风格一致（如 `Kubernetes` 而非 `kubernetes`）
5. **工作经历排序**：按时间倒序排列（最新经历在最上面），但如果同一时间段有多段经历则按开始时间排列
6. **避免重复**：如果新经历中描述的技术栈已经在现有列表中，不要重复添加
7. **事实优先**：基础简历记录事实，不做岗位定制排序、夸张包装或无证据的能力升级；面向 JD 的重组交给 `tailor-resume`
8. **最小修改**：只修改与用户新信息直接相关的段落，避免顺手重写无关经历、调整全篇风格或重排未涉及内容
9. **冲突处理**：新信息与现有内容冲突时，先指出冲突并询问用户确认；不要静默覆盖公司、时间、项目名、职责、证书编号或量化指标
10. **派生文件不联动**：默认只更新 `content/page/about/index-base.md`；不要同步修改 `index.zh-cn.md`、`ai-infra.md` 或其他定制简历，除非用户明确要求

## 完成后

修改文件后告知用户：

- 修改的文件路径
- 更新了哪些部分
- 新增/修改的具体内容摘要
- 未写入或需要用户确认的信息（如果有）
- 是否影响后续定制简历，以及是否建议运行 `tailor-resume`
- 提示：可以进一步使用 `tailor-resume` skill 基于更新后的基础简历生成针对特定岗位的简历
